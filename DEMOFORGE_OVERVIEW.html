Step 1 — Wizard (3 steps)

Name, tagline, description. Source type selection — GitHub Repo (default), Repo + Screenshots, Screenshots only, Live URL. Optional screenshot upload (visual reference only). Style preset (clean/cyber/playful) + aspect ratio (16:9 / 9:16).

Step 2 — Analyze + Generate

Octokit fetches top 25 repo files — always includes README + package.json. Claude extracts stack, features, routes. Then Claude writes a complete custom Remotion TSX file — not templates, real code for that specific app using exact field names from the codebase.

Step 3 — Render + Iterate

Remotion bundles the generated TSX and renders an MP4 to /tmp/, uploaded to Supabase Storage. User can then describe changes in plain English → Claude rewrites the TSX → re-render. Iterations require the user's own Anthropic API key.

The Core: Claude Writes Remotion TSX

Go-Git-It methodology: Before writing a single line of code, Claude extracts exact field names, column headers, button labels, domain terminology, and visual structure from the code snippets. Then it builds app-specific UI components that simulate the real interface — not generic templates.

Animation Patterns

Spring entrance: spring({ frame, fps, config: { damping: 14, stiffness: 120 } })

Staggered rows: Each row delayed by 16 frames, translateX spring-in

Button click: Press (scale 1→0.95→1) + ripple (opacity 0.5→0) over 18 frames

Typing: QUERY.slice(0, Math.floor(frame * 0.35))

Particles: 22 dots orbiting center at frame * 0.013 rad/frame

BYOK — Bring Your Own Key

Free Tier

Each user gets 1 free generation (analyze + generate). The app's own ANTHROPIC_API_KEY is used. A Prisma transaction atomically reserves the slot to prevent TOCTOU races.

API Key Storage

User keys are encrypted with AES-256-GCM using API_KEY_ENCRYPTION_SECRET before writing to user_profiles.anthropicApiKey. The plaintext key never touches the DB.

Iteration Gate

The /iterate endpoint always requires the user's own API key — it never falls back to the app key. If no key is set, the UI shows a prompt to go to Settings.

// userProfile.ts — atomic free generation reservation
await prisma.$transaction(async (tx) => {
  const profile = await tx.userProfile.findUnique({ where: { id: userId } });
  if ((profile?.freeGenerationsUsed ?? 0) >= 1) throw new Error("FREE_GENERATION_EXHAUSTED");
  await tx.userProfile.upsert({
    where:  { id: userId },
    update: { freeGenerationsUsed: { increment: 1 } },
    create: { id: userId, freeGenerationsUsed: 1 },
  });
});

Render Pipeline

POST /api/demos/[id]/render
  ├─ Auth + ownership check
  ├─ 409 if renderStatus === "rendering"  (prevents concurrent render race)
  ├─ DB: renderStatus = "rendering", videoUrl = null
  ├─ return 202 immediately
  └─ void runGeneratedRender()                    // detached background task
       ├─ fs.writeFile("src/remotion/GeneratedDemo.tsx", generatedCode)
       ├─ const { bundle }         = await import("@remotion/bundler")
       ├─ const { renderMedia, … } = await import("@remotion/renderer")
       ├─ bundleLocation = await bundle("src/remotion/generated-entry.ts", { webpackAlias @ → src })
       ├─ composition   = await selectComposition(bundleLocation, "GeneratedDemo", { screenshotUrls })
       ├─ await renderMedia({ codec: "h264", outputLocation: "/tmp/demoforge-gen-{id}.mp4" })
       ├─ fileBuffer = await fs.readFile(outputPath)
       ├─ await supabaseAdmin.storage.from("videos").upload("{userId}/{demoId}/demo.mp4", fileBuffer)
       └─ prisma.update({ renderStatus: "ready", videoUrl })
          on error → prisma.update({ renderStatus: "failed" })

Client: GET /api/demos/[id]/render/status  polled every 2s
  └─ { renderStatus, videoUrl } → shows progress bar → video player on "ready"
Deployment constraint: void runGeneratedRender() requires a persistent Node.js process. Works on Railway/Render/VPS. Killed mid-render on Vercel serverless — use Remotion Lambda for Vercel.

Security Notes

Fixed

  • ✅ Open redirect in /auth/callback — next param validated
  • ✅ TOCTOU on free generation — Prisma transaction
  • ✅ Concurrent render race — 409 guard
  • ✅ Storage orphans — DELETE cleans up files
  • ✅ GeneratedDemo.tsx gitignored — no per-user code committed

Known / Watch

  • ⚠️ No per-user rate limiting on AI endpoints
  • ⚠️ API_KEY_ENCRYPTION_SECRET must be kept secret — rotate if leaked
  • ⚠️ Supabase Storage is public by URL (intentional for sharing)
  • ⚠️ read:org OAuth scope is broader than necessary
  • ⚠️ No Content-Security-Policy header configured
Critical: Never commit .env to version control. The file contains live Supabase credentials, API keys, and the AES encryption secret. Rotate any accidentally exposed secrets immediately in Supabase Dashboard → Database → Reset password.

Key Engineering Decisions

Claude Writes the Entire Video

Instead of templates or a fixed scene library, Claude writes a 1000+ line Remotion TSX file from scratch for every demo. This produces videos that look hand-crafted for that specific app — because they are. The tradeoff: if Claude generates broken TypeScript, the render fails.

Background Render + Polling

void runGeneratedRender() is detached from the HTTP response — 202 returned immediately. Client polls /render/status every 2s. Works on persistent Node (Railway/Render). Fails silently on Vercel serverless — needs Remotion Lambda for that.

Remotion Webpack Externals

@remotion/bundler ships native .node binaries. Both experimental.serverComponentsExternalPackages and webpack.externals must include all four Remotion packages or Next.js webpack crashes at build time.

Prisma 7 + PrismaPg Adapter

Prisma 7 removed the URL from schema.prisma. It lives in prisma.config.ts (CLI) and the PrismaPg({ connectionString }) constructor (runtime). Do NOT install @types/pg separately — it conflicts with the version inside @prisma/adapter-pg.

BYOK with AES-256-GCM

User Anthropic keys are encrypted before DB storage using AES-256-GCM (12-byte random IV, 16-byte auth tag, stored as hex-joined string). The encryption key lives in API_KEY_ENCRYPTION_SECRET — a 64-char hex string that must never be committed.

Atomic Free Generation Gate

A Prisma $transaction wraps the read-check-increment sequence, preventing TOCTOU races where two concurrent requests both pass the "has free gen?" check before either increments the counter.

Deployment

Recommended: Railway / Render / VPS

DemoForge needs a persistent Node.js process for background rendering.

  1. Push to GitHub
  2. Create service from repo
  3. Set all env vars in dashboard
  4. Build: npm run build
  5. Start: node node_modules/next/dist/bin/next start
Local dev: PORT=3001 node node_modules/next/dist/bin/next dev
(npm run dev has a PATH quirk with Remotion native binaries)

Vercel (web only)

Dashboard, wizard, auth, share page — all work fine. The render step will be killed by the serverless timeout.

Move the render pipeline to a dedicated worker: a Railway service receiving jobs via a queue, or Remotion Lambda for AWS-native rendering.
